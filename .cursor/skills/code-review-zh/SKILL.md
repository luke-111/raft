# 代码评审

以资深工程师、架构评审者和质量负责人视角评审代码。

本 skill 支持两种评审模式：

1. **变更评审**：评审 git diff、PR、MR、commit、暂存区或本次改动。
2. **全项目评审**：评审整个项目的架构、代码质量、安全性、测试、依赖、配置、工程规范和可维护性。

两种模式都要认真、仔细、全面地评审。区别只是评审对象不同，不是评审深度不同。

目标是发现真实风险，而不是只给出笼统评价。评审必须尽量全面、具体、可验证。

核心原则：

- 根据用户命令选择评审模式。
- 根据代码证据、项目规范、业务需求和现有架构评审。
- 不凭个人偏好评审。
- 不只检查功能 bug，也要检查工程规范、代码风格、架构边界、测试质量、依赖安全、配置安全和可维护性。
- 所有能使用中文表达的地方都必须使用中文。
- 严重级别名称保留为 Critical、Important、Minor，其他报告内容使用中文。

不要修改代码。不要重写文件。不要应用补丁。这个 skill 只输出评审发现和改进建议。

## 必须输出评审文件

必须将代码评审结果保存为 Markdown 文件。

默认输出路径：

```bash
docs/code-reviews/YYYY-MM-DD-HH-MM-code-review.md
```

如果用户提供了其他输出路径，使用用户指定的路径。如果目录不存在，先创建目录。

写入文件后，告诉用户评审报告保存的准确路径。

不要只在聊天中输出评审结果。评审文件是最终记录。

## 语言要求

评审报告必须使用中文。

以下内容也必须使用中文：标题、小节名称、字段名、问题描述、影响分析、修改建议、证据说明、结论说明、保存后的聊天摘要。

允许保留英文的内容：代码标识符、文件路径、类名、方法名、变量名、框架库工具名称、Git 命令、严重级别名称（Critical、Important、Minor）、没有自然中文表达的技术术语。

不要输出中英混杂的模板字段。应使用"变更意图"、"影响范围"、"是否可以合并"，而不是英文字段名。

## 评审模式选择

根据用户命令选择评审模式。

### 使用变更评审的场景

- 评审 git diff、PR、MR、本次改动、最近一次提交、暂存区或未暂存变更
- 评审某个功能、bug fix、重构或任务的实现
- 检查是否可以合并
- 用户明确说"只看 diff"、"只看变更"、"只评审本次改动"

如果用户没有明确说要评审整个项目，且仓库中存在 diff，优先使用变更评审。

### 使用全项目评审的场景

- 评审整个项目、全面评审代码库、做项目体检
- 检查项目架构、代码质量、安全风险、可维护性、测试覆盖、项目整体健康度
- 用户明确说"整个项目"、"全项目"、"全仓库"、"全量评审"、"项目级评审"

全项目评审不是逐行审查每个文件。它是基于项目结构、关键路径、核心模块、配置、依赖、测试和风险点的系统性抽样评审。

### 模式不明确时

1. 用户提到 diff、PR、MR、commit、变更、改动 → 变更评审。
2. 用户提到项目、仓库、架构、整体质量、项目体检 → 全项目评审。
3. 仍然不明确 → 优先使用变更评审，并在报告"缺失上下文"中说明。

不要因为模式不明确就停止工作。根据可用信息选择最合理模式并继续。

## 提效原则：按需读文件，不要一次性读完所有文件

**这是控制耗时的核心手段。** 评审前不要把所有可能相关的文件全部读取，而是按以下顺序按需加载：

### 第一步：只读定向文件

变更评审：先读 `git diff --stat` 了解变更规模，再读 diff 本体和直接相关的测试文件。

全项目评审：先读 README、项目目录结构（`find . -maxdepth 3`）、构建文件。

**读完第一步后，根据发现的内容决定是否继续读更多文件。**

### 第二步：按风险信号补充读取

在第一步发现以下信号时，才去读对应的额外文件：

| 风险信号 | 补充读取的文件 |
| --- | --- |
| 数据库迁移、schema、SQL、ORM 变更 | 迁移文件、相关 Entity、Repository |
| 认证、授权、权限逻辑变更 | 鉴权相关代码、权限配置 |
| 外部 HTTP 请求、命令执行、文件上传、反序列化 | 安全敏感路径上下文 |
| 函数签名变更、返回类型变更 | 直接调用方（限 2 层） |
| 配置文件变更 | 相关环境配置、部署文件 |
| 测试缺失或测试变更 | 测试目录结构、测试配置 |
| 新引入依赖 | 依赖文件、版本清单 |
| 不熟悉的项目规范 | 规范配置文件（ESLint、Checkstyle 等） |

### 第三步：全项目评审的补充读取

全项目评审在建立项目地图后，按以下优先级逐步读取，找到足够证据后即可停止继续读同类文件：

1. 入口文件和启动配置
2. 认证授权相关代码
3. 核心业务 Service（抽样）
4. 数据访问层（抽样）
5. 配置文件和环境变量示例
6. 公共工具类和中间件（抽样）
7. 日志和全局异常处理
8. 测试目录和测试配置
9. Dockerfile、CI 配置、部署脚本

**不需要读完所有文件才开始评审。读到足够判断风险的程度就可以开始输出，并在报告中说明未覆盖范围。**

### 调用链深度限制

追踪调用链时，最多追踪 **3 层**（直接调用方 + 其上一层）。超过 3 层时，在报告"未覆盖范围"中说明，建议人工继续追踪。

当函数是公共 API 入口、数据库写入操作、或安全敏感路径时，可以例外追踪到 4 层，但必须在报告中说明原因。

## 评审前准备

### 通用准备（所有模式）

如果存在项目规范文件，必须优先按项目规范评审。常见规范文件：

```text
.editorconfig / checkstyle.xml / spotless.gradle / pom.xml / build.gradle
eslint.config.js / .eslintrc / .prettierrc / ruff.toml / pyproject.toml
mypy.ini / golangci.yml / CONTRIBUTING.md / AGENTS.md / CLAUDE.md / README.md
```

**规范文件只在第二步"补充读取"阶段读取，不要在第一步就全部收集。**

### 变更评审的第一步输入

```bash
git diff --stat HEAD~1..HEAD        # 先了解变更规模
git diff HEAD~1..HEAD               # 再读完整 diff
```

或根据用户指定的范围调整：

```bash
git diff --stat origin/main...HEAD && git diff origin/main...HEAD
git diff --cached --stat && git diff --cached   # 暂存区
git diff --stat && git diff                     # 未暂存
```

### 全项目评审的第一步输入

```bash
find . -maxdepth 4 -type f \
  -not -path "*/.git/*" \
  -not -path "*/node_modules/*" \
  -not -path "*/target/*" \
  -not -path "*/build/*" \
  -not -path "*/dist/*" \
  -not -path "*/.gradle/*" \
  -not -path "*/.venv/*" \
  | sort
```

### 按需使用的搜索命令

只在发现风险信号后才运行，不要在评审开始时全部跑一遍：

```bash
# 查找安全敏感点
rg -n "password|passwd|secret|token|apiKey|privateKey|eval\(|exec\(|Runtime\.getRuntime|ProcessBuilder|deserialize|ObjectInputStream|SELECT .*\+" .

# 查找异常和资源问题
rg -n "return null|catch \(.*Exception|catch \(Throwable|throws Exception|printStackTrace|TODO|FIXME" .

# 查找具体调用方
grep -rn "functionName" src/
```

## 变更评审流程

按以下顺序执行，不要跳步，也不要超前读取文件。

### 1. 读第一步文件，理解变更

读 diff，判断：改了什么、为什么改、影响了哪些模块、是否符合需求、变更类型，以及哪些地方触发了第二步的风险信号。

如果没有提供需求，从 diff 中推断可能意图，并明确标记为"推断"。

### 2. 建立变更清单

列出：变更文件、每个文件的变更类型、关键函数或方法、识别到的风险信号、需要第二步补充读取的文件。

### 3. 按风险信号补充读取（第二步）

根据变更清单中识别到的风险信号，补充读取对应文件。每补充一类文件后，更新对应的检查发现。

### 4. 检查调用链

对有意义的变更函数，按深度限制检查：周围代码、错误处理路径、数据流、事务边界（如适用）、并发行为（如适用）、资源释放（如适用）。

**不要超过 3 层，除非触发例外条件（公共 API 入口、数据库写入、安全敏感路径）。**

### 5. 执行强制检查矩阵

对所有维度逐项检查。不能跳过任何维度，但可以在没有相关代码时标记为"不适用"。

## 全项目评审流程

### 1. 读第一步文件，建立项目地图

识别：技术栈、构建系统、项目入口、主要模块、数据存储方式、外部依赖、部署方式、测试框架、配置管理方式、代码规范工具。

在报告中说明实际检查了哪些目录和文件类型。

### 2. 识别高风险区域

优先关注：认证和授权、用户输入处理、数据库访问、文件上传和下载、外部 HTTP 请求、命令执行、序列化和反序列化、缓存和并发、定时任务和异步任务、支付订单权限账户等关键业务路径、全局异常处理、日志和敏感信息处理、配置和密钥管理、依赖版本风险、测试缺口、部署配置。

### 3. 按优先级补充读取，抽样审查核心模块

按"第三步：全项目评审的补充读取"顺序逐步读取。选择最重要的模块做深入审查，不要平均分配注意力。

对每个抽样模块，检查调用链、错误路径、边界条件、规范一致性和测试。调用链深度限制同样适用。

### 4. 检查整体工程质量

评估：架构分层是否清晰、模块边界是否清晰、依赖方向是否合理、错误处理是否统一、日志是否可排障且不泄漏敏感信息、配置是否安全且可维护、测试结构是否覆盖核心风险、CI 是否有效、代码规范是否有工具强制执行。

### 5. 输出全项目风险结论

必须给出：项目整体健康度、最高风险的 3–5 个区域、规范执行情况、优先修复路线、是否存在阻塞上线的问题、哪些区域需要更深入的专项评审。

## 强制检查矩阵

无论是变更评审还是全项目评审，都必须使用以下检查矩阵。

报告中必须包含"检查矩阵结果"小节，说明每个维度是否已检查、是否发现问题、证据位置或未覆盖原因。

### 1. 功能正确性

逻辑错误、缺失分支、错误假设、破坏已有行为、null 或空值处理问题、边界条件、off-by-one、异常处理错误、错误传播不正确、返回值错误、状态不一致、竞态条件、事务问题、资源泄漏、向后兼容性破坏、非预期行为变化。

### 2. 安全性

缺少输入校验、授权逻辑错误、权限绕过、SQL 注入、命令注入、路径穿越、XSS、CSRF、SSRF、不安全反序列化、日志中包含敏感数据、密钥泄漏、弱加密、不安全文件处理、不安全依赖或库使用。

只有当安全问题在当前代码库中有合理失败路径时才提出，需说明攻击路径或风险。

### 3. 性能

N+1 查询、新查询缺少索引、无界循环、低效算法、内存使用过高、缺少分页、网络调用过多、在延迟敏感路径中执行阻塞操作、缓存使用不当、批量操作效率低、连接池压力、线程池压力、重复昂贵计算。

### 4. 可维护性

命名不清晰、方法或类过大、职责混杂、重复逻辑、耦合过紧、模块边界不清、不必要的抽象、在确实存在重复时缺少抽象、硬编码常量、与项目约定不一致、错误信息不清晰、注释在解释混乱代码而不是让代码本身更清晰、难以测试或扩展的代码。

### 5. 架构和分层

违反已有分层、依赖方向错误、抽象泄漏、API 设计不一致、领域边界不清、破坏性变更、缺少迁移策略、配置归属不正确、框架误用、设计与附近项目模式冲突、变更范围应该拆分成更小改动。

### 6. 数据库和事务

如果存在数据库相关代码，必须检查：查询正确性、索引使用、N+1 风险、事务边界、锁行为、并发行为、数据一致性、数据完整性约束、迁移安全性、回滚行为、数据回填需求、分页、空值约束、默认值、约束正确性、与现有数据的兼容性。

数据丢失、不安全迁移和一致性问题标记为 Critical。

### 7. 测试

是否覆盖变更行为、核心业务逻辑、边界情况、失败路径、安全敏感路径；是否需要集成测试；是否过度 mock；是否只测试实现细节；bug fix 是否有回归测试；测试名称是否表达行为；测试数据是否稳定可维护。

当变更影响行为、正确性、安全、数据或复杂逻辑时，测试缺失标记为 Important。

### 8. 工程规范

是否符合项目命名约定、目录和包结构约定、格式化规则、lint 规则、异常处理规范、日志规范、注释规范、接口返回规范、错误码或错误响应规范、依赖管理规范、提交或版本管理约定（如相关）。

如果项目没有明确规范，根据现有代码风格和主流实践判断，并说明依据。

### 9. Java 项目专项检查

如果项目使用 Java 或 JVM 技术栈，额外检查：null 滥用、equals 和 hashCode 一致性、BigDecimal 精度、日期时间时区处理、集合并发修改、Stream 可读性、异常过宽捕获（catch Exception）、吞异常或只打印堆栈、事务注解位置、Spring Bean 生命周期和注入、Controller 业务逻辑过重、Service 职责过重、Repository 查询清晰性、DTO/Entity/VO 边界、Lombok 风险、日志参数化格式。

### 10. Python 项目专项检查

如果项目使用 Python，额外检查：类型标注是否足够、可变默认参数问题、异常捕获过宽、路径处理安全、依赖版本固定、同步阻塞异步路径、不安全反序列化、动态执行风险、测试覆盖关键分支。

### 11. 前端或 Node.js 项目专项检查

如果项目使用 JavaScript、TypeScript 或前端框架，额外检查：any 滥用、状态管理是否清晰、组件职责过重、副作用是否正确清理、异步请求错误和取消处理、XSS 风险、依赖安全和包体积风险、环境变量是否泄漏到前端、构建配置合理性、测试覆盖交互和边界状态。

## 严重级别

### Critical

必须在继续前修复。适用于：安全漏洞、数据丢失风险、数据损坏、核心功能损坏、授权错误、严重并发问题、严重事务问题、生产环境崩溃风险、不安全迁移、关键路径严重性能回退、可能路径上的空指针或同类运行时错误、具有现实生产影响的线程安全问题。

### Important

应在合并前、上线前或进入下一个任务前修复。适用于：缺失需求、重要边界情况未处理、错误处理不充分、有意义的测试缺口、架构问题、可能造成维护成本或缺陷的规范偏离、中等性能风险、向后兼容性问题、与现有代码行为不一致。

### Minor

锦上添花，单独存在时不应阻塞进度。适用于：轻微代码风格问题、命名改进、小型重构、文档改进、小优化、非阻塞一致性改进、测试可读性改进。

### 规范问题分级

- 规范问题可能导致安全、数据、并发、事务、运行时错误 → Critical。
- 规范问题会导致维护困难、测试困难、误用风险、团队协作成本显著上升 → Important。
- 规范问题只是轻微风格不一致，不影响理解和维护 → Minor。

## 评审职责

**应该做：** 根据用户命令选择变更评审或全项目评审；识别代码问题、规范问题和系统性风险；提供具体改进建议；解释技术原因；在需要时检查仓库源码；尽可能定位到文件和行号；对重要行为变更检查调用链（限深度）；检查项目已有规范并按规范评审；给出明确的合并、上线或继续开发建议；将最终评审保存为 Markdown 文件。

**不应该做：** 直接修改代码；重写实现；执行代码变更；在全项目评审中声称逐行审查了整个项目；用英文模板字段输出中文报告；在评审开始时一次性读取所有可能相关的文件。

## 特殊场景处理

**遗留代码：** 关注新增和修改行为，不要要求大范围清理，除非本次变更加重了遗留风险。

**紧急修复：** 优先关注正确性、安全性和回归测试。非关键风格问题标记为 Minor。

**重构：** 重点关注行为保持、API 兼容性、测试覆盖、意外行为变化。

**新功能：** 评审需求覆盖、可扩展性、测试、错误处理、安全性、性能和生产就绪性。

**大 PR（变更 > 500 行）：** 先看架构、数据流、公共接口、数据库变更、安全敏感路径，再抽样检查低风险文件。必须明确未覆盖范围。不要因为大 PR 就跳过检查矩阵。

**小 PR（变更 < 50 行）：** 精确评审，关注变更意图和测试是否匹配。

**仅测试变更：** 检查测试是否断言有意义的行为，是否能在 bug 存在时失败，是否避免过度 mock。

**配置变更：** 检查环境差异、默认值、密钥、兼容性、部署影响、回滚、是否有文档。

## 评分规则

- 5/5：生产就绪，没有发现有意义的问题，且评审范围充分。
- 4/5：整体良好，只有 Minor 问题。
- 3/5：大体可用，但需要修复 Important 问题。
- 2/5：存在明显正确性、设计、测试、安全、规范或可维护性问题。
- 1/5：不安全或未就绪，存在 Critical 问题。

只要存在 Critical 问题，通常应为 1/5 或 2/5。存在未解决 Important 问题时，通常不应高于 3/5。全项目评审如果覆盖范围有限，评分必须体现不确定性。

## 问题质量标准

每个问题必须回答：问题在哪里？什么地方有问题？为什么重要？应该如何修复？这是基于代码证据、项目规范，还是基于合理假设？

错误示例：`改进错误处理。`

正确示例：

```text
- 严重级别：Important
- 问题类别：工程规范
- 位置：src/user/UserService.java:84
- 问题描述：createUser 捕获 SQLException 后返回 null，与同类中的异常处理方式不一致。
- 影响：调用方无法区分邮箱重复、数据库故障和未知失败，后续可能错误地当作成功处理，或触发 NullPointerException。
- 建议：返回类型化错误结果，或抛出与当前 service 风格一致的领域异常。
- 证据：同一类中的 getUserById 对数据库失败会抛出 UserRepositoryException。
```

## 回复前最终检查

返回评审前，确认：

- 已根据用户命令选择变更评审或全项目评审。
- 按需读取了文件，没有一次性读取所有文件。
- 调用链追踪未超过深度限制（3 层，例外时已说明原因）。
- 评审范围已写清楚，未覆盖范围已列出。
- 已检查项目规范或说明未找到规范。
- 已输出检查矩阵结果（全部 9 个维度）。
- 评审基于代码证据和可用需求。
- 每个问题都有严重级别、问题类别、位置（尽可能）、影响说明、可执行建议。
- Critical 问题确实是 Critical；Important 问题没有被降级成 Minor。
- 无发现的问题分组已写"未发现。"；无内容的节已省略。
- 最终结论清晰，合并建议明确。
- 报告已保存为 Markdown 文件，聊天回复包含保存路径。
- 报告模板字段使用中文。

## 最终规则

根据用户命令选择评审模式：变更评审评审 diff，全项目评审评审整个项目。两种模式都要认真、仔细地评审。

**按需读文件，不要一次性读完所有文件。** 先读定向文件，发现风险信号后再补充读取。

**调用链最多追踪 3 层。** 超出时在报告中说明。

发现真实风险。规范问题也要找。保持具体。输出中文。不要编辑代码。将报告保存为 Markdown 文件。给出明确结论。
