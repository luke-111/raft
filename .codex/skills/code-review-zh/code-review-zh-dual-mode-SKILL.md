# 代码评审

以资深工程师和架构评审者的视角评审代码。

本 skill 支持两种评审模式：

1. Diff 评审：评审 git diff、PR、MR 或本次变更。
2. 全项目评审：评审整个项目的架构、代码质量、安全性、测试、依赖、配置和可维护性。

目标是判断代码是否正确、安全、可维护、经过适当测试，并给出明确的风险和改进建议。

核心原则：根据用户命令选择评审范围；根据代码证据和需求进行评审，不凭个人偏好评审。

不要修改代码。不要重写文件。不要应用补丁。这个 skill 只输出评审发现和改进建议。

## 必须输出评审文件

必须将代码评审结果保存为 Markdown 文件。

默认输出路径：

```bash
docs/code-reviews/YYYY-MM-DD-HH-MM-code-review.md
```

如果用户提供了其他输出路径，使用用户指定的路径。

如果目录不存在，先创建目录。

写入文件后，告诉用户评审报告保存的准确路径。

不要只在聊天中输出评审结果。评审文件是最终记录。

## 评审模式选择

根据用户命令选择评审模式。

### 使用 Diff 评审的场景

当用户请求包含以下含义时，使用 Diff 评审：

- 评审 git diff
- 评审 PR 或 MR
- 评审本次改动
- 评审最近一次提交
- 评审 staged 或 unstaged changes
- 评审某个功能、bug fix、重构或任务的实现
- 检查是否可以合并
- 用户明确说“只看 diff”、“只看变更”、“review changes”

如果用户没有明确说要评审整个项目，且仓库中存在 diff，优先使用 Diff 评审。

### 使用全项目评审的场景

当用户请求包含以下含义时，使用全项目评审：

- 评审整个项目
- 全面评审代码库
- 做项目体检
- 检查项目架构
- 检查代码质量
- 检查安全风险
- 检查可维护性
- 检查测试覆盖
- 检查项目整体是否健康
- 用户明确说“整个项目”、“全项目”、“全仓库”、“whole project”、“entire repo”、“full codebase”

全项目评审不是逐行审查每个文件。它是基于项目结构、关键路径、核心模块、配置、依赖、测试和风险点的系统性抽样评审。

### 模式不明确时

如果用户命令不明确：

1. 如果用户提到 diff、PR、MR、commit、changes，使用 Diff 评审。
2. 如果用户提到项目、仓库、架构、整体质量，使用全项目评审。
3. 如果仍然不明确，优先使用 Diff 评审，并在报告的“缺失上下文”中说明评审范围是根据当前可见 diff 推断的。

不要因为模式不明确就停止工作。根据可用信息选择最合理模式并继续。

## 何时使用

在以下场景使用此 skill：

- 评审 git diff
- 评审 Pull Request 或 Merge Request
- 评审已完成的实现任务
- 合并前评审代码
- 评审 bug fix
- 评审重构
- 评审生成代码或 Agent 编写的代码
- 检查实现是否符合计划、工单、规格说明或需求
- 评审整个项目的架构、质量、安全、测试和可维护性
- 对项目做系统性风险扫描

当用户说“代码评审”、“review this diff”、“check this PR”、“detect bugs”、“review my changes”、“评审整个项目”、“项目体检”、“检查全仓库”等类似请求时，使用此 skill。

## 评审心态

评审时遵循这些原则：

- 客观：基于代码证据、需求和项目约定提出问题。
- 建设性：提供具体、可执行的改进建议。
- 教育性：当问题原因不明显时，解释为什么重要。
- 完整性：覆盖正确性、安全性、性能、可维护性、架构、数据库行为和测试。
- 精确性：尽可能引用具体文件和行号。
- 审慎但公平：不要编造在当前代码库中不现实的风险。
- 严重级别准确：不要把代码风格问题标记为 Critical。
- 范围清晰：明确本次是 Diff 评审还是全项目评审。

## Diff 评审输入

Diff 评审前，尽可能收集以下信息：

- Git diff
- PR 或 MR 描述
- 工单、计划、规格说明或需求
- 变更的源文件
- 修改行附近的未修改代码
- 被修改函数或方法的直接调用方
- 被修改函数或方法调用的下游函数或方法
- 相关测试
- 项目已有约定
- 相关数据库迁移、schema、SQL 查询和 ORM 映射
- 相关配置变更

如果关键上下文缺失，仍然基于已有证据继续评审，并在报告中明确列出缺失的上下文。

## 全项目评审输入

全项目评审前，尽可能收集以下信息：

- README、架构文档、开发文档
- 项目目录结构
- 构建文件和依赖文件，例如 pom.xml、build.gradle、package.json、pyproject.toml、go.mod
- 入口文件和启动配置
- 核心业务模块
- 公共库、工具类、中间件、拦截器、过滤器
- API 层、Service 层、Repository 或 DAO 层
- 数据库迁移、schema、SQL、ORM 映射
- 配置文件和环境变量示例
- 鉴权、认证、权限控制相关代码
- 日志、错误处理和异常处理机制
- 测试目录和测试配置
- CI 配置
- Dockerfile、部署脚本、Helm chart 或其他部署配置
- 安全敏感代码，例如文件上传、命令执行、外部请求、反序列化、加密、登录注册、支付、权限判断

全项目评审必须说明抽样范围。不要声称已经逐行审查整个项目，除非实际完成了逐行审查。

## 常用 Git 命令

### Diff 评审命令

评审最近一次提交：

```bash
git diff --stat HEAD~1..HEAD
git diff HEAD~1..HEAD
```

评审相对于 main 的变更：

```bash
git fetch origin
git diff --stat origin/main...HEAD
git diff origin/main...HEAD
```

评审已暂存变更：

```bash
git diff --cached --stat
git diff --cached
```

评审未暂存变更：

```bash
git diff --stat
git diff
```

查看变更文件列表：

```bash
git diff --name-only origin/main...HEAD
```

### 全项目评审命令

查看项目结构：

```bash
find . -maxdepth 3 -type f | sort
```

排除常见构建产物和依赖目录：

```bash
find . -maxdepth 4 -type f   -not -path "*/.git/*"   -not -path "*/node_modules/*"   -not -path "*/target/*"   -not -path "*/build/*"   -not -path "*/dist/*"   -not -path "*/.gradle/*"   -not -path "*/.idea/*"   | sort
```

查找核心配置：

```bash
find . -maxdepth 4 -type f \(   -name "README*" -o   -name "pom.xml" -o   -name "build.gradle" -o   -name "settings.gradle" -o   -name "package.json" -o   -name "pyproject.toml" -o   -name "go.mod" -o   -name "Dockerfile" -o   -name "*.yml" -o   -name "*.yaml" \) | sort
```

查找潜在安全敏感点：

```bash
rg -n "password|secret|token|apiKey|privateKey|eval\(|exec\(|Runtime\.getRuntime|ProcessBuilder|deserialize|ObjectInputStream|SELECT .*\+|TODO|FIXME" .
```

调查调用点时，使用仓库搜索工具，例如 grep、rg 或语言感知搜索工具。

## Diff 评审流程

当评审模式是 Diff 评审时，按顺序执行以下流程。

### 1. 理解变更

判断：

- 改了什么
- 为什么改
- 它看起来要解决什么业务目标或技术目标
- 影响了哪些模块、分层、API、任务、数据模型或配置文件
- 是否符合已说明的需求
- 是否超出了已说明的需求范围
- 这次变更是 bug fix、新功能、重构、仅测试变更、配置变更、迁移，还是混合变更

如果没有提供需求，从 diff 中推断可能意图，并明确标记为“推断”。

### 2. 检查影响范围

对每个有意义的变更函数、方法、API、查询、模型、配置或迁移：

- 检查周围代码。
- 检查直接调用方。
- 检查直接下游调用。
- 检查相关测试。
- 检查错误处理路径。
- 检查数据流和状态变化。
- 如果适用，检查事务边界。
- 如果适用，检查并发行为。
- 如果适用，检查资源归属和释放。

当行为依赖周围代码时，不要只评审改动行本身。

### 3. 检查变更风险

重点检查功能正确性、安全性、性能、可维护性、架构、数据库和测试。使用后文统一检查清单。

## 全项目评审流程

当评审模式是全项目评审时，按顺序执行以下流程。

### 1. 建立项目地图

先识别：

- 技术栈和语言
- 构建系统
- 项目入口
- 主要模块
- 核心业务路径
- 数据存储方式
- 外部依赖
- 部署方式
- 测试框架
- 配置管理方式

在报告中说明实际检查了哪些目录和文件类型。

### 2. 识别高风险区域

优先检查：

- 认证和授权
- 用户输入处理
- 数据库访问
- 文件上传和下载
- 外部 HTTP 请求
- 命令执行
- 序列化和反序列化
- 缓存和并发
- 定时任务和异步任务
- 支付、订单、权限、账户、数据导入导出等关键业务路径
- 全局异常处理
- 日志和敏感信息处理
- 配置和密钥管理

### 3. 抽样审查核心模块

选择最重要的模块做深入审查，而不是平均分配注意力。

优先顺序：

1. 入口和路由层
2. 认证授权层
3. 核心业务 Service
4. 数据访问层
5. 配置和中间件
6. 测试覆盖
7. 部署和 CI

对每个抽样模块，检查调用链、错误路径、边界条件和测试。

### 4. 检查整体工程质量

评估：

- 架构分层是否清晰
- 模块边界是否清晰
- 依赖方向是否合理
- 错误处理是否统一
- 日志是否可排障且不泄漏敏感信息
- 配置是否安全且可维护
- 测试结构是否覆盖核心风险
- CI 是否能阻止明显问题进入主干
- 项目是否容易被新成员理解和修改

### 5. 输出全项目风险结论

全项目评审必须给出：

- 项目整体健康度
- 最高风险的 3 到 5 个区域
- 优先修复路线
- 是否存在阻塞上线或阻塞继续开发的问题
- 哪些区域需要更深入的专项评审

## 统一检查清单

无论是 Diff 评审还是全项目评审，都使用以下检查清单。

### 功能正确性

重点寻找：

- 逻辑错误
- 缺失分支
- 错误假设
- 破坏已有行为
- null 或空值处理问题
- 边界条件问题
- off-by-one 问题
- 异常处理错误
- 错误传播不正确
- 返回值错误
- 状态不一致
- 竞态条件
- 事务问题
- 资源泄漏
- 向后兼容性破坏
- 非预期行为变化

优先指出真实缺陷，而不是代码风格问题。

### 安全性

重点寻找：

- 缺少输入校验
- 授权逻辑错误
- 权限绕过
- SQL 注入
- 命令注入
- 路径穿越
- XSS
- CSRF
- SSRF
- 不安全反序列化
- 日志中包含敏感数据
- 密钥泄漏
- 弱加密
- 不安全文件处理
- 不安全依赖或库使用

只有当安全问题在当前代码库中有合理失败路径时才提出。需要说明攻击路径或风险。

### 性能

重点寻找：

- N+1 查询
- 新查询缺少索引
- 无界循环
- 低效算法
- 内存使用过高
- 缺少分页
- 网络调用过多
- 在延迟敏感路径中执行阻塞操作
- 缓存使用不当
- 批量操作效率低
- 连接池压力
- 线程池压力
- 重复昂贵计算

说明该问题在现实规模下为什么重要。

### 可维护性

重点寻找：

- 命名不清晰
- 方法或类过大
- 职责混杂
- 重复逻辑
- 耦合过紧
- 模块边界不清
- 不必要的抽象
- 在确实存在重复时缺少抽象
- 硬编码常量
- 与项目约定不一致
- 错误信息不清晰
- 注释在解释混乱代码，而不是让代码本身更清晰
- 难以测试或扩展的代码

不要对紧急修复或很小的改动过度打磨，除非可维护性风险确实明显。

### 架构

重点寻找：

- 违反已有分层
- 依赖方向错误
- 抽象泄漏
- API 设计不一致
- 领域边界不清
- 破坏性变更
- 缺少迁移策略
- 配置归属不正确
- 框架误用
- 设计与附近项目模式冲突
- 变更范围应该拆分成更小改动

除非有明确理由，否则优先遵循已有架构和本地代码约定。

### 数据库

如果数据库代码、schema、SQL 查询、迁移、Repository 或 ORM 映射发生变化，检查：

- 查询正确性
- 索引使用
- N+1 风险
- 事务边界
- 锁行为
- 并发行为
- 数据一致性
- 数据完整性约束
- 迁移安全性
- 回滚行为
- 数据回填需求
- 分页
- 空值约束
- 默认值
- 约束正确性
- 与现有数据的兼容性

数据丢失、不安全迁移和一致性问题应标记为 Critical。

### 测试

评估测试是否：

- 覆盖变更行为
- 覆盖核心业务逻辑
- 覆盖边界情况
- 覆盖失败路径
- 在适用时覆盖安全敏感路径
- 当单元测试不足时覆盖集成行为
- 测试真实行为，而不是只验证 mock
- 避免过度 mock
- 如果没有实现，测试会失败
- 能防止回归
- 测试公开行为，而不是实现细节

当变更影响行为、正确性、安全、数据或复杂逻辑时，测试缺失应标记为 Important。

## 严重级别

使用以下固定严重级别。

### Critical

必须在继续前修复。

适用于：

- 安全漏洞
- 数据丢失风险
- 数据损坏
- 核心功能损坏
- 授权错误
- 严重并发问题
- 严重事务问题
- 生产环境崩溃风险
- 不安全迁移
- 关键路径上的严重性能回退
- 可能路径上的空指针或同类运行时错误
- 具有现实生产影响的线程安全问题

### Important

应在合并前、上线前或进入下一个任务前修复。

适用于：

- 缺失需求
- 重要边界情况未处理
- 错误处理不充分
- 有意义的测试缺口
- 架构问题
- 可能导致缺陷的可维护性问题
- 中等性能风险
- 向后兼容性问题
- 与现有代码行为不一致
- 有实际影响的框架或设计模式误用

### Minor

锦上添花。单独存在时不应阻塞进度。

适用于：

- 代码风格问题
- 命名改进
- 小型重构
- 文档改进
- 小优化
- 非阻塞一致性改进
- 测试可读性改进

## 评审职责

应该做：

- 根据用户命令选择 Diff 评审或全项目评审。
- 识别代码问题和风险。
- 提供具体改进建议。
- 解释技术原因。
- 在需要时检查仓库源码。
- 尽可能定位到文件和行号。
- 对重要行为变更检查完整调用链。
- 在有帮助时推荐工具、测试或后续检查。
- 给出明确的合并、上线或继续开发建议。
- 将最终评审保存为 Markdown 文件。

不应该做：

- 直接修改代码。
- 重写实现。
- 执行代码变更。
- 替人类做最终技术决策。
- 替代人工评审批准。
- 除非用户明确要求，不要提供大段替换代码。
- 除非变更与历史代码交互，不要评审无关遗留代码。
- 除非实际运行过测试或看到证据，不要声称测试通过。
- 在全项目评审中声称逐行审查了整个项目，除非确实逐行审查。

## 特殊场景处理

### 遗留代码

关注新增和修改行为。不要要求大范围清理遗留代码，除非本次变更加重了遗留风险，或依赖了不安全的遗留行为。

### 紧急修复

优先关注正确性、安全性和回归测试。除非风格或大范围重构造成真实风险，否则应标记为 Minor。

### 重构

重点关注行为保持、API 兼容性、测试覆盖、简化效果和意外行为变化。

### 新功能

评审需求覆盖、设计适配性、可扩展性、测试、错误处理、安全性、性能和生产就绪性。

### 大 PR

先看架构、数据流、公共接口、数据库变更、安全敏感路径和关键逻辑。然后抽样检查低风险文件。

### 小 PR

精确评审。小 diff 应该有清晰意图、最小范围和适当测试。

### 仅测试变更

检查测试是否断言有意义的行为，是否避免过度 mock，是否覆盖边界情况，以及是否能在 bug 或缺失行为存在时失败。

### 配置变更

检查环境差异、默认值、密钥、兼容性、部署影响、回滚和配置是否有文档。

## 报告文件格式

保存的 Markdown 报告必须使用以下中文结构。输出内容也必须使用中文。

```markdown
# 代码评审报告

## 评审概览

- 评审模式：Diff 评审 / 全项目评审
```

## 评分规则

谨慎使用整体评分。

- 5/5：生产就绪，没有发现有意义的问题。
- 4/5：整体良好，只有 Minor 问题。
- 3/5：大体可用，但需要修复 Important 问题。
- 2/5：存在明显正确性、设计、测试、安全或可维护性问题。
- 1/5：不安全或未就绪，存在 Critical 问题。

只要存在 Critical 问题，通常应为 1/5 或 2/5。

存在未解决 Important 问题时，通常不应高于 3/5。

全项目评审如果覆盖范围有限，评分必须体现不确定性，不要给出过度自信的高分。

## 问题质量标准

每个问题必须回答：

- 问题在哪里？
- 什么地方有问题？
- 为什么重要？
- 应该如何修复？
- 这是基于代码证据，还是基于假设？

错误示例：

```text
改进错误处理。
```

正确示例：

```text
- 严重级别：Important
- 位置：src/user/UserService.java:84
- 问题描述：createUser 捕获 SQLException 后返回 null。
- 影响：调用方无法区分邮箱重复、数据库故障和未知失败，后续可能错误地当作成功处理，或者触发 NullPointerException。
- 建议：返回类型化错误结果，或抛出与当前 service 风格一致的领域异常。
- 证据：同一类中的 getUserById 对数据库失败会抛出 UserRepositoryException。
```

## 回复前最终检查

- 每个问题都有严重级别。
- 每个问题尽可能有位置。
- 每个问题解释了影响。
- 每个问题有可执行建议。
- Critical 问题确实是 Critical。
